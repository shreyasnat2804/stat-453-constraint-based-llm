% ---- Abstract ----

\section*{Abstract}
Small language models ($\sim$1B parameters) offer practical advantages in cost and inference speed over large frontier models, yet they consistently underperform on multi-constraint instruction-following tasks. We present a complete end-to-end pipeline---dataset preprocessing, supervised fine-tuning via full parameter updates and LoRA, evaluation via deterministic rule-based checking and an LLM judge, and k-fold cross-validation---designed to quantify and improve constraint-following in 1B-parameter models using the RECAST-30K dataset. We compare full fine-tuning against LoRA (rank 8, applied to attention projections) on Llama 3.2 1B Instruct, using a custom constraint-aware loss that augments the standard cross-entropy objective with a per-batch constraint satisfaction penalty. We evaluate on six constraint types: length, keyword, start-with, end-with, style, and topic, measuring both per-constraint pass rates and the hard Constraint Satisfaction Rate (hard CSR). Our pipeline establishes baselines, fine-tunes under both regimes, and enables systematic per-category ablation to identify where each training strategy provides the most benefit.

\section{Introduction and Motivation}

Large language models are powerful but expensive to deploy at scale. Models in the 1--2B parameter range offer a practical alternative with substantially lower compute cost and faster inference latency, yet they struggle with \textit{constraint-following}: simultaneously satisfying explicit requirements on format, length, tone, style, keywords, and structure imposed by a prompt. This difficulty compounds as the number of simultaneous constraints grows---a setting that directly mirrors real-world deployment scenarios such as drafting structured reports, composing constrained marketing copy, or generating formatted API responses.

Benchmarks such as IFEval~\cite{zhou2023ifeval} and FollowBench~\cite{jiang2024followbench} have documented these failures comprehensively. Llama-3.2-1B-Instruct achieves only $\sim$55--60\% prompt-strict accuracy on IFEval, illustrating how severely 1B-scale models struggle with even moderate constraint complexity.

The RECAST framework~\cite{guo2025recast} addresses the data side of this problem by synthesizing multi-constraint training sets where every constraint has an automated validator---enabling fine-grained, constraint-level supervision. RECAST fine-tuning at 7--8B scale outperforms much larger instruct-tuned models, but it has not been tested at 1B scale. Separately, LoRA~\cite{hu2022lora} and full fine-tuning have been studied in the context of reasoning and code, but not specifically for constraint-following, leaving open whether LoRA's low-rank updates are expressive enough for multi-dimensional constraint adherence at small scale~\cite{shuttleworth2024lora}.

\textbf{Research questions.} This project addresses three open questions:
\begin{enumerate}[noitemsep,topsep=2pt]
    \item Can RECAST-based fine-tuning meaningfully improve constraint-following in 1B-parameter models?
    \item Does full fine-tuning outperform LoRA for this task, or do low-rank updates suffice?
    \item Which constraint categories benefit most from fine-tuning, and which remain difficult?
\end{enumerate}

\section{Related Work}

\textbf{Constraint-following benchmarks.} IFEval~\cite{zhou2023ifeval} introduced a structured evaluation suite covering 25 verifiable instruction categories. FollowBench~\cite{jiang2024followbench} extended this with multi-level difficulty constraints, showing sharp degradation as constraint count increases. Both benchmarks diagnose the problem but do not provide training recipes.

\textbf{RECAST.} Guo et al.~\cite{guo2025recast} propose a constraint back-translation framework that synthesizes multi-constraint training data by extracting constraints from existing high-quality responses and injecting them back into prompts, guaranteeing physical achievability. Every constraint has an automated validator (rule-based or LLM-based), enabling fine-grained training signal. At 7B and 8B scales, RECAST fine-tuning outperforms larger instruct-tuned models; however, behavior at 1B parameters is unexplored.

\textbf{LoRA vs.\ full fine-tuning.} Hu et al.~\cite{hu2022lora} introduced Low-Rank Adaptation, which freezes the base model and injects trainable rank-$r$ matrices into attention projections, training only $\sim$0.5\% of parameters. Shuttleworth et al.~\cite{shuttleworth2024lora} show that full fine-tuning adjusts existing singular vectors more conservatively while LoRA introduces ``intruder dimensions'' that can accelerate catastrophic forgetting---but this analysis is for reasoning and code tasks. Whether these dynamics hold for constraint-following remains open.

\textbf{Instruction-tuning data.} The Tulu corpus~\cite{ivison2023tulu} assembles diverse instruction-following examples from FLAN, ShareGPT, OpenAssistant, and other sources, providing broad coverage needed to prevent catastrophic forgetting when fine-tuning on the narrower RECAST constraint distribution.

\subsection{Data Preprocessing}

The raw RECAST-30K JSONL is processed by an eight-stage cleaning pipeline
(\texttt{src/crllm/dataset/\allowbreak{}preprocess/\allowbreak{}preprocess.py}):

\begin{enumerate}[noitemsep,topsep=2pt]
    \item \textbf{Language detection.} Non-English prompts are filtered using \texttt{langdetect} (ISO-639-1 check on the first 2000 characters of \texttt{winner\_prompt}).
    \item \textbf{HTML and entity removal.} Tags and entities are stripped from prompt and response fields with compiled regex patterns.
    \item \textbf{Emoji and control character removal.} Emojis are replaced with spaces (\texttt{emoji} library, with fallback regex); control characters, invisible Unicode, and long symbol runs are removed.
    \item \textbf{Quality gates.} Prompts shorter than 15 words, predominantly non-printable text ($<85\%$ printable), or with empty responses are dropped.
    \item \textbf{Constraint field integrity.} Each record's \texttt{added\_constraint} dict is validated: keys must map to non-empty string lists, and the stated \texttt{added\_constraint\_num} must match the actual count.
    \item \textbf{Stopword removal.} Applied only to the prompt (not the response) after all quality gates pass, to avoid artificially shortening prompts before the length gate.
    \item \textbf{Exact deduplication.} SHA-256 fingerprints of normalized prompts detect exact duplicates.
    \item \textbf{Near-deduplication.} MinHash LSH (\texttt{datasketch}, Jaccard threshold $\tau = 0.85$, 128 permutations) filters near-duplicate prompts. MinHashes are pre-computed in parallel via \texttt{joblib}; the sequential LSH insertion phase follows.
\end{enumerate}

A post-cleaning distribution audit flags constraint categories whose share falls below 50\% of the expected uniform share, enabling early detection of imbalance before training.

The pipeline processes the dataset in parallel (default 8 workers) and produces a clean JSONL at \texttt{datasets/recast\_30k\_clean.jsonl} ready for fine-tuning.

\subsection{Model and Dataset}

\textbf{Models.} The primary model for this study is \textbf{Llama 3.2 1B Instruct} (1.24B parameters), a publicly available instruction-tuned causal language model. The proposal identified four candidate models; Llama 3.2 1B Instruct was prioritized for initial experiments given its widespread use as a small-model benchmark.

\textbf{RECAST-30K.} The training data is the cleaned RECAST-30K split. The dataset covers at least 12 constraint categories: Style, Topic, Length, Keyword, End\_With, Start\_With, Language, Helpfulness, Format, Tone, Role-Playing, and Numerical. Each record provides:
\begin{itemize}[noitemsep, topsep=2pt]
    \item \texttt{winner\_prompt} --- the multi-constraint instruction;
    \item \texttt{response\_of\_winner\_prompt} --- the winning response that satisfies the constraints;
    \item \texttt{added\_constraint} --- a dict mapping constraint category to a list of natural-language constraint descriptions;
    \item \texttt{rule\_evaluate\_dict} --- a structured dict encoding the rule-based parameters (word length bounds, required keywords with counts, start/end strings) used for automated scoring.
\end{itemize}

\subsection{Full Fine-Tuning with Constraint-Aware Loss}

The full fine-tuning script (\texttt{src/crllm/training/\allowbreak{}full\_finetune/\allowbreak{}train\_full\_finetune.py}) makes all model parameters trainable and uses the HuggingFace \texttt{Trainer} with gradient checkpointing. The pipeline:

\begin{enumerate}[noitemsep, topsep=2pt]
    \item Formats each record as a user/assistant chat turn using the model's \texttt{chat\_template} (falling back to a plain \texttt{User:}/\texttt{Assistant:} prefix if unavailable).
    \item Tokenizes to \texttt{max\_length=512} and masks prompt tokens (\texttt{labels=-100}) so loss is computed only on the response.
    \item Pre-computes a \textbf{constraint score} per example from the gold response against the four rule-based constraints (length, keyword, start-with, end-with), yielding a float in $[0, 1]$.
\end{enumerate}

\textbf{Constraint-aware loss.} The custom \texttt{ConstraintAwareTrainer} adds a constraint penalty to the cross-entropy loss:
\begin{equation}
    \mathcal{L}_{\text{total}} = \mathcal{L}_{\text{CE}} + \lambda \cdot \mathcal{L}_{\text{constraint}}
    \label{eq:loss}
\end{equation}
where $\mathcal{L}_{\text{constraint}} = 1 - \overline{s}$, and $\overline{s}$ is the mean constraint score over the batch. When gold responses fully satisfy all rule constraints ($\overline{s} = 1$), $\mathcal{L}_{\text{constraint}} = 0$ and only the CE loss contributes. Batches with constraint-violating examples push $\mathcal{L}_{\text{constraint}}$ toward 1, increasing the total loss and biasing the optimizer toward constraint-satisfying outputs. The default $\lambda = 0.1$ was chosen to keep the penalty term sub-dominant; setting $\lambda = 0$ recovers standard CE-only training.

A custom \texttt{ConstraintDataCollator} extends \texttt{DataCollatorForSeq2Seq} to pass the pre-computed constraint scores through the batch as a tensor, enabling the trainer to access them during the loss computation.

\textbf{Hyperparameters.}
\begin{itemize}[noitemsep, topsep=2pt]
    \item Optimizer: AdamW; learning rate: $2\times10^{-5}$; cosine LR scheduler with 3\% warmup
    \item Effective batch size: 32 (per-device batch 4, gradient accumulation 8)
    \item Epochs: 2; max sequence length: 512 tokens
    \item Precision: bfloat16 on A100 40GB; gradient checkpointing enabled
    \item Validation split: 5\%; best checkpoint saved by lowest \texttt{eval\_loss}
\end{itemize}

\subsection{LoRA Fine-Tuning}

The LoRA pipeline (\texttt{src/crllm/training/lora\_finetune/lora\_base.ipynb}, implemented as a Colab notebook) applies rank-$r$ adapter matrices to the query and value projection matrices (\texttt{q\_proj}, \texttt{v\_proj}) of all attention layers:
\begin{equation}
    W' = W + \frac{\alpha}{r} \cdot B A
    \label{eq:lora}
\end{equation}
where $W \in \mathbb{R}^{d \times d}$ is frozen, $A \in \mathbb{R}^{r \times d}$ and $B \in \mathbb{R}^{d \times r}$ are trainable, $r=8$ (sweep: also $r=16$), $\alpha = 2r$, and dropout $= 0.05$. This yields $\approx 0.5\%$ trainable parameters (roughly 6.2M of 1.24B).

The LoRA pipeline uses \texttt{trl.SFTTrainer} with the same prompt template and dataset loading logic. The Llama 3 chat format is applied explicitly:
\begin{lstlisting}
<|begin_of_text|><|start_header_id|>user<|end_header_id|>
{instruction}<|eot_id|>
<|start_header_id|>assistant<|end_header_id|>
{response}<|eot_id|>
\end{lstlisting}

\textbf{Hyperparameters.}
\begin{itemize}[noitemsep, topsep=2pt]
    \item Learning rate: $10^{-4}$; cosine scheduler; 10 warmup steps; weight decay 0.01
    \item Per-device batch: 8; gradient accumulation: 4 (effective batch: 32)
    \item Epochs: 1; max sequence length: 512; precision: bfloat16
    \item Only the LoRA adapter ($\sim$30--60 MB) is saved; the base model remains frozen
\end{itemize}

\subsection{Evaluation: Constraint Validation Script}

The evaluation pipeline (\texttt{src/crllm/evaluation/\allowbreak{}validate\_constraints.py}) runs in two modes:

{\sloppy\textbf{Rule-based mode (default).} Four constraints are evaluated deterministically
from the \texttt{rule\_evaluate\_dict} field:\par}

\begin{table}[H]
\centering
\small
\begin{tabular}{lp{9cm}}
\toprule
\textbf{Constraint} & \textbf{Check} \\
\midrule
\texttt{length}     & Word count of response falls within $[\text{min}, \text{max}]$ derived from \texttt{word\_length.func\_input} \\
\texttt{keyword}    & Every required keyword appears $\geq$ required count (case-insensitive) \\
\texttt{start\_with} & Response (stripped, lowercased) begins with the required string \\
\texttt{end\_with}   & Response (stripped, lowercased) ends with the required string \\
\bottomrule
\end{tabular}
\caption{Rule-based constraint checks. Each returns 0 (fail), 1 (pass), or None (not applicable).}
\label{tab:rule_checks}
\end{table}

\textbf{LLM-judge mode.} When \texttt{--judge\_model} is provided, the judge model receives the response plus plain-English descriptions of all applicable constraints and outputs a JSON verdict \texttt{\{"length": 0/1, "keyword": 0/1, ...\}}. This extends coverage to the two qualitative constraints: \textbf{style} and \textbf{topic} (sourced from \texttt{added\_constraint\_from\_LLM}). The default judge is a 7B instruct model (e.g., \texttt{Qwen/Qwen2.5-7B-Instruct}); Gemma-2-9B-Instruct is also supported via the standalone \texttt{judge.py} module with optional 4-bit quantization. If the judge output is unparseable, the example falls back to rule-based scoring.

\textbf{Scoring.}
\begin{equation}
    \text{CSR}^{(i)} = \frac{\text{constraints satisfied in example } i}{\text{constraints checked in example } i}
\end{equation}
\begin{equation}
    \text{Hard CSR} = \mathbb{1}\left[\text{all checked constraints satisfied}\right]
\end{equation}
\begin{equation}
    \text{Overall loss} = 1 - \frac{\sum_i \text{satisfied}_i}{\sum_i \text{total}_i}
\end{equation}

Results are written to CSV (one row per example) and a companion JSON summary with per-constraint pass rates.

\subsection{Modular Constraint Checker}

The standalone \texttt{ConstraintChecker} class (\texttt{src/crllm/evaluation/constraint\_checker.py}) provides a dispatch-table architecture supporting 18 constraint types:

\begin{table}[H]
\centering
\small
\begin{tabular}{ll}
\toprule
\textbf{Category} & \textbf{Constraint Types} \\
\midrule
Length & \texttt{word\_count}, \texttt{sentence\_count}, \texttt{paragraph\_count} \\
Keywords & \texttt{existence}, \texttt{frequency}, \texttt{forbidden} \\
Structural & \texttt{start\_with}, \texttt{end\_with} \\
Capitalization & \texttt{all\_caps\_count}, \texttt{all\_lowercase} \\
Format & \texttt{bullet\_points}, \texttt{numbered\_list}, \texttt{sections}, \texttt{json} \\
Language & \texttt{english} \\
Punctuation & \texttt{no\_comma} \\
Detectable & \texttt{postscript}, \texttt{highlight} \\
\bottomrule
\end{tabular}
\caption{Constraint types supported by \texttt{ConstraintChecker}.}
\label{tab:checker}
\end{table}

Each handler returns \texttt{True}/\texttt{False}/\texttt{None} (None for inapplicable constraints). The \texttt{check\_all} method aggregates results into \texttt{per\_\allowbreak{}constraint\_csr} and \texttt{hard\_csr}.

\subsection{Cross-Validation}

A k-fold cross-validation pipeline (\texttt{Cross\_validation/\allowbreak{}cross\_\allowbreak{}validation\_kfold.ipynb}) partitions the RECAST training split into $k$ folds, trains on $k-1$ folds, and evaluates on the held-out fold. This provides a more robust estimate of generalization performance than a single train/validation split and enables variance estimation across folds. Parallel Colab notebooks implement LoRA and full fine-tuning variants within the same k-fold framework.

\subsection{Evaluation Protocol}

\begin{enumerate}[noitemsep, topsep=2pt]
    \item \textbf{Zero-shot baseline.} Generate responses from the pretrained Llama 3.2 1B Instruct checkpoint on a 500-example subset of the cleaned RECAST test split.
    \item \textbf{Fine-tuned evaluation.} Load the fine-tuned checkpoint (full FT or LoRA adapter merged with base), generate responses on the same 500 examples, and score with the validation script.
    \item \textbf{LLM judge.} For a 200-example subset, run the LLM judge to score style and topic in addition to the four rule-based constraints.
    \item \textbf{Per-category analysis.} Compare per-constraint pass rates and CSR gains between training regimes.
    \item \textbf{IFEval generalization.} Evaluate on the IFEval benchmark to detect regression in general instruction-following ability after fine-tuning.
\end{enumerate}

\subsection{Constraint-Aware Loss Ablation}

The custom loss (Equation~\ref{eq:loss}) was motivated by the observation that RECAST gold responses do not uniformly satisfy all four rule-based constraints---approximately 26\% of constraints fail in the baseline. By down-weighting batches with higher constraint violation rates, $\mathcal{L}_{\text{constraint}}$ provides a signal complementary to the CE loss. Training logs surface both \texttt{ce\_loss} and \texttt{constraint\_loss} separately at each logging step, enabling inspection of their relative magnitudes over the course of training.

\section{Discussion}

\subsection{Key Design Decisions}

\textbf{Constraint scoring on gold responses.} Pre-computing constraint scores from the gold responses rather than from model outputs avoids the inference overhead of running the model twice per step. It treats constraint satisfaction as a data-quality weight rather than a reward signal, which is appropriate for supervised fine-tuning where the gold response is already the target.

\textbf{Prompt masking.} Masking prompt tokens in the labels tensor ($\texttt{labels} = -100$) ensures the CE loss is computed only over the response portion. This is standard practice in instruction-tuning but critical for RECAST: the constraint specifications embedded in prompts are often long, and including them in the loss would dilute the gradient signal from the response.

\textbf{Two-level deduplication.} The preprocessing pipeline uses exact SHA-256 deduplication followed by MinHash LSH near-deduplication (Jaccard $\geq 0.85$). The two-stage design is deliberate: exact deduplication is $O(1)$ and has zero false positives; fuzzy deduplication catches paraphrased near-duplicates that exact matching misses, at the cost of a configurable false-positive rate.

\textbf{LoRA target modules.} Applying LoRA only to \texttt{q\_proj} and \texttt{v\_proj} (query and value projections) is a standard choice that captures the attention reweighting most relevant to instruction following while keeping the adapter small ($\sim$30--60 MB vs.\ 2+ GB for the full model).

\subsection{Limitations}

\begin{enumerate}[noitemsep, topsep=2pt]
    \item \textbf{Rule-based coverage.} The deterministic evaluator covers only 4 of 12+ constraint categories. Style, tone, role-playing, and other qualitative constraints require the LLM judge, which adds inference cost and can produce unparseable outputs (falling back to rule-based scoring for those examples).
    \item \textbf{Scale.} This study focuses on Llama 3.2 1B Instruct. The proposal targeted four 1--2B models; resource constraints narrowed the primary experiments to one model.
    \item \textbf{Catastrophic forgetting.} We did not mix Tulu into the training data as proposed. Without general instruction-following data, fine-tuning risks degrading IFEval performance. This trade-off has not yet been measured.
    \item \textbf{LLM judge bias.} Using a 7B model as judge for evaluating 1B model outputs introduces a potential systematic bias: the judge may apply different standards than a human annotator.
\end{enumerate}

% ---- Section 7 ----
\section{Conclusion}

We have built a complete end-to-end pipeline for studying multi-constraint instruction-following in small language models, covering data preprocessing, two fine-tuning regimes (full parameter updates with constraint-aware loss and LoRA with SFTTrainer), hybrid evaluation (deterministic rule-based plus LLM judge), and k-fold cross-validation for robustness estimation. The pipeline is modular: each stage is independently runnable, and the constraint checker supports 18 constraint types extendable via the dispatch-table architecture.

Baseline evaluation on Llama 3.2 1B Instruct shows an overall constraint pass rate of $\sim$74\% on a rule-based subset of RECAST, with length adherence as the weakest category ($\sim$62\%). The constraint-aware loss term provides a principled mechanism to signal batch-level constraint quality to the optimizer beyond standard CE loss. Full experimental results comparing LoRA and full fine-tuning---and measuring the benefit of the constraint-aware loss term---will complete the evaluation.

Future work includes extending the study to all four proposed 1--2B models (Llama 3.2 1B, Gemma 3 1B, Qwen2.5 1.5B, SmolLM2 1.7B), mixing Tulu data to prevent catastrophic forgetting, and developing the constraint-aware text editor stretch goal.
